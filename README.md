# Prototype-PM Skill 使用说明

## 1. Skill 简介

`prototype-pm` 是一套面向产品经理的原型设计标准化 Skill，用于帮助产品经理在 **App、Web SaaS / 后台系统、桌面客户端** 等场景中，生成风格一致、结构清晰、具备真实产品质感的原型页面。

该 Skill 的核心目标是解决产品原型中常见的以下问题：

- 页面框架不统一
- 原型视觉风格不稳定
- 信息层级不清晰
- 页面留白、圆角、阴影、交互规范不一致
- AI 生成结果容易被参考图片风格带偏
- Web、App、客户端之间缺少统一的设计迁移规则

## 2. 适用对象

本 Skill 适合以下角色使用：

- 产品经理
- 交互设计师
- UI/UX 设计师
- 使用 AI / Codex / Claude 生成原型的团队成员
- 需要统一原型设计规范的产品团队

## 3. 适用场景

可用于生成、评审或标准化以下类型的产品原型：

- Web SaaS 系统
- B 端后台管理系统
- CRM / 客户管理系统
- 数据看板 / Dashboard
- 企业管理平台
- 桌面客户端软件
- App / 移动端 / 小程序原型
- AI 助手工作台类页面

## 4. 核心设计原则

### 4.1 先识别平台，再执行设计

在生成任何原型前，必须先判断目标平台：

- Web SaaS / 后台 / Dashboard：使用多列工作台布局
- 桌面客户端：保持 Web 信息密度，并增强窗口缩放、标题栏、状态栏、折叠面板意识
- App / 移动端：转换为移动端结构，不直接压缩桌面端布局
- 平台不明确：默认按 Web SaaS / 后台处理

### 4.2 参考图片只用于理解，不用于复制

当用户提供截图、竞品页面、参考图或视觉样例时，只允许用于理解：

- 业务语义
- 信息结构
- 功能优先级
- 交互意图

不得复制参考图的：

- 视觉风格
- 色彩方案
- 组件外观
- 页面节奏
- 装饰手法
- 特殊渐变或特效

最终输出必须遵循本 Skill 的原始设计方法。

## 5. 推荐目录结构

建议将 Skill 文件组织为以下结构：

```txt
prototype-pm/
├── SKILL.md
├── README.md
└── reference/
    └── source-default-layout.html
```

说明：

- `SKILL.md`：Skill 主规则文件
- `README.md`：安装说明与使用指南
- `reference/source-default-layout.html`：原始参考布局文件，用于沉淀默认页面结构与设计 Token

## 6. 安装方式

将整个 `prototype-pm` 文件夹放入支持 Skill 的工具目录中。

常见目录示例：

```txt
.codex/
└── skills/
    └── prototype-pm/
        ├── SKILL.md
        ├── README.md
        └── reference/
            └── source-default-layout.html
```

安装后，在对应 AI / Codex 环境中调用该 Skill，即可用于原型生成、原型评审和原型规范化。

## 7. 使用方式

### 7.1 生成 Web 后台原型

示例 Prompt：

```txt
请使用 prototype-pm Skill，帮我生成一个 CRM 客户管理后台首页原型。
要求：
1. 使用 Web SaaS 多列工作台布局；
2. 包含左侧导航、组件面板、主工作区和 AI 助手面板；
3. 使用真实模拟数据；
4. 保持高级、克制、专业、可信赖的视觉风格；
5. 输出完整 HTML。
```

### 7.2 生成 App 原型

示例 Prompt：

```txt
请使用 prototype-pm Skill，帮我生成一个移动端销售助手首页原型。
要求：
1. 不要直接复制桌面端四列布局；
2. 转换为移动端结构；
3. 包含顶部上下文、模块切换、KPI 卡片、主内容流、AI 助手入口和底部 Tab；
4. 输出完整 HTML。
```

### 7.3 评审已有页面

示例 Prompt：

```txt
请使用 prototype-pm Skill，从布局、间距、圆角、阴影、信息层级、交互状态和响应式适配角度，评审这个页面是否符合原型基础框架规范，并给出修改建议。
```

## 8. 输出要求

使用该 Skill 生成原型时，应满足以下要求：

1. 必须生成完整首屏，不允许中间工作区为空。
2. 页面结构优先使用 CSS Grid。
3. 组件内部优先使用 Flexbox。
4. 未提供业务数据时，需要自动生成真实感模拟数据。
5. HTML 输出应包含语义化结构、自包含 CSS 和必要 JavaScript。
6. 每个区域只保留一个主操作，其他操作应弱化为次级按钮。
7. 避免廉价渐变、强玻璃拟态、高饱和颜色、厚重阴影和 PPT 式页面。
8. 页面文字必须可读，避免溢出、重叠、截断和阴影被裁切。

## 9. 视觉风格关键词

该 Skill 推荐的整体视觉气质为：

- 高级
- 克制
- 专业
- 科技感
- 温暖
- 可信赖
- 清晰
- 数据驱动
- 真实产品感

应避免：

- 廉价感
- 过度渐变
- 颜色刺眼
- 信息拥挤
- 留白不足
- 卡片过平
- 图标风格不统一
- 字体层级混乱
- 像 PPT 而不像真实系统
- 像模板站而不像专业产品

## 10. 关键检查清单

在最终输出前，需要检查：

- 是否已识别平台类型：App、Web 或桌面客户端
- 页面内边距是否符合 22px / 18px / 14px 节奏
- 主卡片圆角是否控制在 22px 到 26px
- 阴影是否柔和，是否避免黑重阴影
- Hover 是否不会改变卡片高度或破坏布局
- 折叠侧边栏是否保留恢复入口和 tooltip
- 移动端是否完成结构转换，而不是简单压缩桌面布局
- 参考图片是否只用于理解业务，而没有覆盖本 Skill 的设计方法

## 11. 维护建议

如需继续扩展该 Skill，建议优先补充：

- 更多 Web SaaS 页面模板
- App 首页 / 列表页 / 详情页模板
- 桌面客户端窗口规范
- 表格、筛选、搜索、卡片、图表组件规范
- 典型业务场景 Prompt 示例
- 评审用检查表

扩展时应保持一个原则：

> 新增内容可以补充业务场景，但不得破坏本 Skill 的基础布局、视觉 Token、交互节奏和参考图片独立原则。

- 支持 App / Web / 桌面客户端场景识别

- 支持统一视觉 Token
- 支持统一圆角、阴影、间距和 Hover 规则
- 支持参考图片独立约束
- 支持原型生成与设计评审
